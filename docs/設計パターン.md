# Claude Code — 状況別コンテキスト自動ロードのパターン

Claude Codeで「特定の状況でのみ特定のルール・ドキュメントを読み込む」仕組みの設計パターンをまとめる。

---

## 背景と問題意識

CLAUDE.mdに `@.claude/rules/*.md` を列挙すると、会話開始時に全ルールが静的にロードされる。
git操作をしない会話でもworktree.mdが読み込まれるなど、コンテキストの無駄遣いが発生する。

---

## アプローチ 1: Hooks（イベント駆動）

### 概要

ツール呼び出しのイベントに反応してシェルコマンドを実行し、その出力をClaudeが読み取る仕組み。
**push型** — Claudeの判断に依存せず、強制的にコンテキストを注入できる。

### 設定場所

`.claude/settings.json` の `hooks` フィールド。

### トリガーできるイベント

| イベント | タイミング |
|---------|-----------|
| `PreToolUse` | ツール実行前 |
| `PostToolUse` | ツール実行後 |
| `Notification` | Claude から通知が発生したとき |
| `Stop` | Claude が応答を停止したとき |

### 「状況」の検出方法

- **ツール名で判断**: Bash / Edit / Write / Read のどれが呼ばれたか
- **コマンド内容で判断**: Bash コマンドに `git` `supabase` `vercel` などが含まれるか
- **ファイルパスで判断**: Edit/Write の対象が `src/app/` 配下なら Next.js ルールを注入、など

### 向いているケース

- 絶対に守らせたいルール（git操作、DB操作、環境変数の扱いなど）
- 読み忘れが致命的な手順（PRマージ後のworktree削除など）

### 注意点

- フックの出力が増えすぎるとそれ自体がコンテキストを圧迫する
- 誤検知（関係ないコマンドでも発火）に注意

## アプローチ 2: CLAUDE.mdのメタ指示（Claude自身の判断）

### 概要

「〜の状況になったら〜のファイルを読め」という指示をCLAUDE.mdに書き、
ClaudeがReadツールで自律的に参照する仕組み。
**pull型** — Claude自身が状況を判断してコンテキストを取りに行く。

### 設定場所

`CLAUDE.md` に自然言語で記述する。

```markdown
## ルール参照ガイド

以下の状況になったら、対応するルールファイルを必ずReadツールで読むこと：

| 状況 | 読むファイル |
|------|------------|
| git操作・PR作成・ブランチ操作 | `.claude/rules/git.md` |
| worktree の作成・削除 | `.claude/rules/worktree.md` |
| Supabase のテーブル変更・マイグレーション | `.claude/rules/supabase.md` |
| 環境変数の追加・変更 | `.claude/rules/env.md` |
| Vercel へのデプロイ・CI確認 | `.claude/rules/vercel.md` |
```

### 向いているケース

- コードなしで設定したい
- ルールの種類が多く、毎回全部読む必要がない補助的なドキュメント
- 状況の定義があいまいなもの（「相談っぽい会話のとき」など）

### 注意点

- Claudeの判断に依存するため、読み忘れリスクがある
- Hooksと組み合わせて「重要なものはHooks、補助的なものはメタ指示」に分けると良い

## アプローチ 3: MCPツール（ツール駆動）

### 概要

ルールや設計ドキュメントを「呼び出せるツール」として公開し、
Claudeが必要と判断したときに能動的に取得する仕組み。
**pull型** — MCPサーバーを介してオンデマンドにコンテキストを取得する。

### 仕組み

1. MCPサーバーに `get_rules(context_type: string)` のようなツールを定義
2. Claudeが「supabaseを操作しようとしている」と判断したら `get_rules("supabase")` を呼び出す
3. サーバーが対応するルールファイルの内容を返す

### Hooksとの違い

| | Hooks | MCP |
|--|-------|-----|
| トリガー | システムが強制 | Claudeが自律的に判断 |
| 型 | push | pull |
| 設定コスト | 低〜中 | 高 |
| 柔軟性 | ツールイベントに限定 | 任意のタイミング |

### 向いているケース

- ルールが多く動的に管理したい
- 外部システム（Notionなど）のドキュメントをリアルタイムで参照させたい
- チーム開発でルールを中央管理したい

### 注意点

- MCPサーバーの構築・運用コストが高い
- 個人プロジェクトでは過剰になりやすい

---

## 比較まとめ

| アプローチ | トリガー | 確実性 | 設定コスト | 向いている用途 |
|-----------|---------|--------|-----------|--------------|
| Hooks | ツールイベント | 高（強制） | 低〜中 | 絶対守らせたいルール |
| メタ指示 | Claudeの判断 | 中 | 低 | 補助的なドキュメント |
| MCP | Claudeの判断 | 中 | 高 | 大規模・チーム開発 |

---

## このプロジェクトへの適用案

現状の `@.claude/rules/*.md` を全ロードから以下に移行する：

**Hooksで強制ロード（重要度：高）**
- `git.md` — Bashコマンドに `git` が含まれるとき
- `worktree.md` — `git worktree` が含まれるとき
- `env.md` — Edit/WriteでEnvファイルを操作するとき

**メタ指示でClaudeに任せる（重要度：中）**
- `supabase.md` — Supabase操作の話題になったとき
- `vercel.md` — デプロイ・CI確認の話題になったとき
- `docs.md` — Issue実装前のドキュメント更新が必要なとき

**常時ロードを維持（軽量なもの）**
- `github.md` — PR操作は頻度が高く短いため常時ロードでも問題ない
