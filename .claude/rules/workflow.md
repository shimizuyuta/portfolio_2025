# 作業原則

## 会話コンテキストより実際の状態を優先する

会話の記憶（「さっきこうだった」）ではなく、必ずツールで現在の状態を確認してから行動する。

悪い例：「さっき feat/xxx ブランチで作業していたからまだオープンのはず」→ そのまま操作
良い例：`gh pr view <番号>` で確認 → マージ済みと判明 → main から切り直す

## ブランチ作成前のチェックリスト

1. `gh pr list` でオープン PR を確認
2. 関連する PR は `gh pr view <番号>` でマージ済みかを確認
3. `git log origin/main --oneline -5` で main の最新を確認
4. worktree で作業ディレクトリを作成する（→ `worktree.md` 参照）

並行開発は必ず **git worktree** を使う。`git checkout -b` は使わない。
worktree を使うことでブランチごとに独立したディレクトリが作られ、変更が混ざらない。

## Issue 作成前のヒアリング（必須）

Issue を作成する前に、以下を確認してからユーザーの承認を得る：

- 目的・背景（なぜやるか）
- 実装方針（どうやるか、技術的な選択肢など）
- 受け入れ条件（何ができたら完了か）

これらが未確定の場合は、壁打ちを先に行ってから Issue を作成する。
ユーザーが「Issue を作りたい」と言った場合でも、上記が明示されていなければヒアリングを行う。

## 実装着手前の確認（必須）

Issue の実装を開始する前に、以下がすべて確定していることを確認する：

- [ ] 何を作るか（機能・構造）が確定している
- [ ] どう見せるか（デザイン・内容・文言）が確定している
- [ ] ユーザーの承認を得た

「ページを作る」と「ページの中身を決める」は別のステップ。
構造が決まっても、内容・デザインが未確定なら壁打ちを先に行う。

## 作業の進め方（こまめな確認）

- 複数の操作をまとめて実行しない
- 1アクション（Issue 作成・ファイル編集・PR 作成など）ごとにユーザーに報告し、次の指示を待つ
- 「〜しました。次に〜をしてよいですか？」の形で確認を取る

## ビルド確認の方針

PR 作成前に以下を**この順番で**実行し、すべてパスしてから PR を作成する：

```bash
npm run lint          # Biome によるコード品質チェック
npx tsc --noEmit      # TypeScript 型チェック（Biome は型エラーを検出しない）
```

- `npm run build` はローカルでは不要（時間がかかるため）
- Vercel のビルド検証は CI に任せる
- **ただし CI が通るまでユーザーに PR レビューを依頼しない**
  - PR 作成後に `gh pr checks <PR番号>` で CI の結果を確認する
  - CI が通過してから初めてユーザーに報告する
  - CI が落ちた場合は自力で修正してから報告する

## PR のスコープ

1 PR には1つの関心事のみ含める。異なる種類の変更は必ず別 PR に分ける。

悪い例：ルール変更（`.claude/rules/`）と機能実装（`src/`）を同じ PR に含める
良い例：ルール変更のみの PR・機能実装のみの PR をそれぞれ作成する

代表的な PR の種類（混在させない）：
- `.claude/` の設定・ルール変更
- アプリケーションコードの実装
- スタイル・デザイン変更
- ドキュメント更新

## Issue 単位のチェックリスト

ユーザーへの報告時は、対象 Issue の進捗をチェックリストで表示する。

フォーマット例：
```
### Issue #XX <タイトル>
- [x] 完了したステップ
- [ ] 次のステップ
- [ ] 残りのステップ
```
