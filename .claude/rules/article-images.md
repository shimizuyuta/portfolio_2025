# 記事画像ルール（生成・配置）

**いつ読むか:** 記事の図・イラスト・アイキャッチを生成／配置する作業の前。

このファイルは記事本文に入れる図の「作り方」と「置き方」を定める。
メディア方針・記事規約は `blog.md`、サムネ生成は `.claude/commands/make-thumbnail.md`。

---

## 原則（blog.md と整合）

- 本文には `【画像: 説明】` プレースホルダのみを置く。
- AI は画像の**素材（PNG）を生成してよい**。ただし **本文への差し込み・アップロード（`thumbnail_url` や本文への URL 反映）は人間が `/admin` から行う**。**AI は本文に画像 URL を埋めない。**
- 生成物は素材であり、公開可否・掲載位置の最終判断は人間。

## 生成

- ツールは `scripts/gemini-image/gen.py`（既定モデル `gemini-3-pro-image`）を使う。日本語ラベル入りの図もほぼ崩れない。**`gemini-2.5` 系は日本語が崩れるので使わない。**
- **配色はサイトのブランドに合わせる**（青／ネイビー系。primary `#0050b5` / accent `#026fd7`。オレンジ・赤・緑・ティールは使わない）。ブランド色の正は `scripts/thumbnail/theme.ts` と `src/app/globals.css`。
- 描く日本語ラベルは**本文の表記と一致**させる。制度・数値・固有名詞は本文・一次情報と食い違わせない。
- 図中に**実在の個人情報を入れない**（利用者名・事業所名等は `〇〇` 等の架空・抽象データ）。ブランドロゴ・URL は焼き込まない。
- スマホ閲覧主体のため、**文字大きめ・高コントラスト**。

## 配置（重要）

- **記事ごとにフォルダを分ける:** `tmp/記事画像/<slug>/`
- **ファイル名は、差し込む `【画像: …】` の「…」の文言と完全一致させる。**
  - 例: 本文が `【画像: 個別支援計画の作成フロー（アセスメントからモニタリングまで）を示した図】` なら
    `tmp/記事画像/individual-support-plan-ai/個別支援計画の作成フロー（アセスメントからモニタリングまで）を示した図.png`
  - こうすると、どの画像がどのプレースホルダに入るか一目で対応づく。
- `tmp/` は `.gitignore` 済み。**画像素材は git に載せない。**
- 生成の試作・比較は `tmp/記事画像/<slug>/` 内や `tmp/gemini-image/out/` で行い、確定版のみ上記の文言名で置く。

## 画像化しないもの

- 単純な**表**（列見出しが明確なデータの一覧）は、画像より本文の Markdown 表が望ましい（SEO・スマホ・アクセシビリティで有利）。「表の例」は画像化を避け、本文に表を書くことを検討する。

## 手順の目安

1. 対象記事の本文から `【画像: …】` を洗い出す（公開記事は `/api/articles?slug=` で取得）。
2. 各図の内容を本文に忠実に設計し、ブランド配色でプロンプトを書く（`scripts/gemini-image/prompt.example.txt` を雛形に）。
3. `scripts/gemini-image/gen.py` で生成し、目視で日本語崩れ・内容の正確さを確認。
4. 確定版を `tmp/記事画像/<slug>/<プレースホルダ文言>.png` に置く。
5. 差し込み・アップロードは人間に引き継ぐ（本文へ URL は埋めない）。
