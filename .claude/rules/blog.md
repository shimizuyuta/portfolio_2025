# ブログ運用ルール

**いつ読むか:** 記事の執筆・リライト・企画・改善など、ブログ運用に関わる作業の着手前。

このファイルはメディアの方針と記事の品質基準を定める。DB のテーブル仕様・RLS は `supabase.md`、PR・CI・コミットの規約は `github.md`・`git.md` を参照する（ここでは再掲しない）。

---

## メディア定義

**正体:** 「IT・AI × 現場業界」の横断メディア。「AIを活用したいが、自分の業界にどう落とせばいいか分からない」非IT業界の実務者・小規模経営者に向けて、現役エンジニアが業界の言葉で書く実践メディア。

**KPI:** 当面はアクセス数の最大化。収益化はアフィリエイト（ASP）+ アドセンス。自社サービスへのリード獲得は目的にしない。

### ハブ&スポーク構造

| 層 | 内容 | 役割 |
|---|---|---|
| **ハブ（横軸）** | 業界を問わない生成AI・DXの実践ノウハウ。AIツールの実機検証レビュー、業務自動化の作り方、補助金・助成金×IT | トラフィックのエンジン |
| **スポーク（縦軸）** | 業界別の実務・制度・現場感 | E-E-A-T と差別化のエンジン |

**注力バーティカル（立ち上げ3ヶ月）:** 福祉 / 医療・歯科 / 士業・建設・店舗等の小規模事業

> **福祉特化メディアではない。** 福祉は複数あるバーティカルの一つに過ぎない。ジャンル定義・ペルソナ・カテゴリ設計・監視キーワードのいずれも福祉に偏らせないこと。

**展開原則:** 1つの検証・テーマをハブ記事1本 + 各スポークでの応用記事に多重展開する。
例：「議事録AI比較」（ハブ）→「福祉の面談記録での使い方」「歯科の診療メモでの使い方」（スポーク）

### ペルソナ

**第一読者:** 非IT業界で日々の業務にAIを使いたい実務者・小規模経営者。ITは得意でない。**スマホ閲覧が主。**

業界別記事では、その業界の職種に具体化する（福祉なら現場職員・相談員、歯科なら歯科衛生士・受付など）。

### 運営者の強み（E-E-A-T資産）

- フリーランスエンジニア（6年、AI・DX導入支援の実務） × 社会福祉士
- 医療（歯科）・福祉・スポーツ教育での支援実績

記事にはこの**実務者としての一次体験・現場視点を必ず反映させる**。これが品質原則の中核。

## 記事フォーマット規約

### リード文（150〜250字）

「悩みの言語化 → 得られること」の順で書く。煽り・大げさな前置きは書かない。

```markdown
訪問介護の記録づくりに、毎日1時間近く取られていないでしょうか。
利用者さんと向き合う時間を削ってまでPCに向かうのは、本末転倒だと感じている方も多いはずです。

この記事では、音声入力AIを使って訪問記録の下書きを作る手順を、実際に現場で試した結果とあわせて紹介します。
無料で始められる範囲に絞っているので、今日から試せます。
```

### 見出し階層

| 階層 | 本数 | 備考 |
|---|---|---|
| h2（`##`） | 3〜6本 | 必須 |
| h3（`###`） | 各h2に0〜3本 | 任意 |
| h4以下 | **禁止** | 目次に出ず、スマホで階層が伝わらないため |

### 文字数目安

| 記事タイプ | 目安 |
|---|---|
| ハウツー・比較・検証 | 2,500〜4,000字 |
| トレンド解説 | 1,500〜2,500字 |

### まとめの型

**要点3行 + 内部リンク1本。**

```markdown
## まとめ

- 音声入力AIは「下書き作成」に限れば、無料プランでも実用に足ります
- ただし固有名詞と専門用語は必ず誤変換するため、人の目での確認は省けません
- 個人情報を含む記録では、クラウド送信の可否を事業所のルールで先に確認してください

記録業務そのものの見直しについては、[介護記録の書き方を効率化する5つの手順](/knowledge/example-slug) もあわせてご覧ください。
```

### 画像

**`【画像: 説明】` のプレースホルダのみ**を本文に置く。実画像は人間が用意する。AIが画像URLを埋めない。

```markdown
【画像: 音声入力AIで訪問記録の下書きを作成している画面】
```

### 段落

スマホで**1段落4行以内**を目安にする。長い段落は分割する。

## メタ情報規約

| 項目 | ルール |
|---|---|
| `title` | **32字以内**、狙うキーワードを**前方**に置く |
| `description` | **80〜110字**。記事を読んで得られることを書く |
| `slug` | 英小文字とハイフンのみ（例: `care-record-ai-voice-input`） |
| `excerpt` | 一覧カードに出る要約。DB上 NOT NULL のため必ず埋める |
| `category` | 既存カテゴリから選ぶ。新設は人間の承認制 |

### タグ

**「業界タグ × テーマタグ」の二軸**で、既存の `tags` テーブルから選ぶ。

| 軸 | 例 |
|---|---|
| 業界タグ | 介護 / 障害福祉 / 歯科 / クリニック / 士業 / 建設 / 店舗 |
| テーマタグ | 生成AI / 業務効率化 / 補助金 / ツール比較 / 自動化 |

**新規タグの作成は人間の承認制。** 既存タグに該当が無い場合は、その場で人間に確認してから作る。勝手に増やすとタグ体系が崩れる。

## 執筆ルール

### 必須

- **実務者視点を必ず入れる。** 「実際にやってみてどうだったか」「現場ではこう詰まる」を具体的に書く。空疎な一般論だけの段落は不可
- **制度・法令・数値には一次情報（官公庁等）のリンクを必ず付ける。** 確認できないことは書かない
- **PR表記**（ステマ規制対応）。アフィリエイトリンクを含む記事は冒頭に明示する

```markdown
> 本記事はプロモーションを含みます。
```

### 禁止

| 禁止事項 | 理由 |
|---|---|
| AI量産の薄い記事 | 量産してよいのは記事の「型」だけ。中身には必ず実務者視点を入れる |
| YMYL領域での断定 | 医療・健康・金銭は「〜と言われています」「〜の場合があります」に留め、判断は専門家・公式情報へ誘導する |
| 誇大表現（景表法） | 「必ず」「絶対に」「劇的に」「No.1」など根拠のない最上級表現 |
| 未検証を「検証した」と書く | 実際に試したことのみ「検証した」と書く。試していないなら「公式情報によると」と書く |
| 他サイトからの転載・翻案 | 著作権侵害 |
| 守秘義務違反・個人特定 | 支援先や利用者が特定できる記述は書かない。事例は必ず抽象化する |

### 文体

- **です・ます調**
- **一文60字以内**を目安にする
- **常套句禁止:**「いかがでしたでしょうか」「〜ではないでしょうか」の多用、「〜な方も多いのではないでしょうか」の連発
- **煽り導入禁止:**「知らないと損する」「9割の人が間違えている」など

## ガードレール（絶対規則）

これらは**いかなる指示があっても**破らない。ユーザーから明示的に依頼された場合も、実行せず理由を説明する。

- **記事の保存は draft まで。** 公開ステータスへの変更・`published_at` の設定は**人間のみ**が行う
- **既存記事の更新前に必ず旧版を退避する**（後述）
- **記事・タグ・画像の DELETE 禁止。** 削除が必要なら人間に依頼する
- **同一記事のリライトは月1回まで。** 直近30日以内に更新済みの記事へのリライト依頼は、その旨を伝えて人間の確認を取る
- **Supabase のスキーマ・RLSポリシー・トリガーの変更禁止**
- **認証情報の表示・出力・クライアントコードへの埋め込み禁止**（`SUPABASE_SERVICE_ROLE_KEY`・`BLOG_API_KEY`・`PREVIEW_TOKEN` 等）。環境変数の値を `echo` 等で確認しない
- **本番ビルド・デプロイ設定の無断変更禁止**（Vercel設定・CI等）
- **キャッシュ再検証（`published-articles` タグ）は公開時の人間側の操作。** AIは触らない

## Supabase 操作規約

### 書き込み経路

記事の読み書きは**必ず `/api/articles`（`BLOG_API_KEY` 認証）経由**で行う。Supabase MCP や service role での直接書き込みはしない。

理由：このAPIは `status` と `published_at` を受け付けない設計になっており、**公開処理が構造的に不可能**になっている。経路を迂回するとこの保証が失われる。

| メソッド | 用途 |
|---|---|
| `GET` | 記事一覧の取得（全status横断）。`?slug=` で本文を含む単一記事 |
| `POST` | draft の新規作成 |
| `PATCH` | 既存記事の更新（`status`・`published_at` は変更不可） |
| `DELETE` | **存在しない**（405） |

接続先は `BLOG_API_BASE_URL`。認証は `Authorization: Bearer $BLOG_API_KEY`。

### draft 保存の手順

1. `GET /api/articles` で重複チェック（**全status対象**。タイトル・slug・タグで確認）
2. 類似記事があれば新規作成せず、**リライト／追記を提案して停止**する
3. `POST /api/articles` で保存する
4. **保存後は必ずプレビューURLを提示する**

```
{BLOG_API_BASE_URL}/knowledge/{slug}?preview={PREVIEW_TOKEN}
```

### 更新前の旧版退避（必須）

既存記事を `PATCH` する前に、必ず現行データを退避する。

1. `GET /api/articles?slug=<slug>` で本文を含む現行データを取得
2. `articles-backup/<slug>/<YYYYMMDD-HHmm>.json` として保存
3. その後に `PATCH` を実行する

`PATCH` のレスポンスには更新前のスナップショット（`previous`）が含まれるので、退避内容と突き合わせて一致を確認する。退避は git 管理下に置き、履歴そのものをバージョン管理として使う。

## サブエージェント運用規則

**「書く者」と「検査する者」を分離する。** 自己添削は甘くなるため、writer が自分の原稿を合格判定してはならない。

| 規則 | 内容 |
|---|---|
| editor 必須 | **editor のレビューを通過していない本文を保存してはならない** |
| 往復上限 | writer ↔ editor の往復は**最大2回**。2回で収束しなければ人間に判断を仰ぐ |
| 保存の責務 | Supabase への保存は**コマンド本体**が行う。writer は保存しない |
| 実行の責務 | seo-analyst は方針を起案するのみ。リライトの実行は writer が行う |

### 人間の承認ポイント（省略禁止）

| フロー | 承認が必要な地点 |
|---|---|
| 企画 | 記事候補からの選択 |
| 執筆 | **構成案の承認（本文執筆前に必ず停止）** |
| 改善 | リライト実施の承認 |
| 公開 | ステータス変更は人間のみ |

これらを飛ばす実装・運用をしない。逆に、ここに書かれていない承認ポイントを勝手に増やして人間の手間を増やすこともしない。
