# ラッコキーワード データ取り込み仕様

## 置き場所

人間がラッコからエクスポートしたファイルを、次の場所に置く：

```
tmp/rakko/<テーマ or slug>/
```

- `tmp/` は `.gitignore` 済み（キーワードデータはリポジトリに載せない）
- フォルダ名は発掘時はテーマ名、執筆時は記事の slug を推奨
- ファイル形式は CSV / JSON どちらでもよい。エージェントはヘッダ（日本語）を見て解釈する

## フェーズ別：何をエクスポートするか

### 発掘フェーズ（/trend-scan → trend-scout）
候補の「需要」と「勝てるか」を数値で裏付けるために使う。

| ラッコの画面 | 得られるもの | パイプラインでの用途 |
|---|---|---|
| キーワード一覧（月間検索数・SEO難易度つき） | KWごとの検索ボリューム・難易度 | 候補の数値スコアリング（ボリューム×難易度） |
| サジェスト / 関連キーワード | 関連語リスト | スポークネタ・副キーワード・h2候補 |
| 獲得キーワード調査（競合URL） | 競合が流入を得ている語 | 競合が取れている語の逆算（任意） |

### 執筆フェーズ（/write-article → writer / editor）
構成の「網羅性」と「意図適合」を上げるために使う。

| ラッコの画面 | 得られるもの | パイプラインでの用途 |
|---|---|---|
| 見出し抽出 | 上位記事の H2/H3 | 競合が触れる論点のカバー＋抜けの発見 |
| 共起語 | 頻出関連語 | 本文の網羅性チェック（自然に触れているか） |
| Q&A検索 / LSI・PAA | 実際の疑問・関連質問 | h2・FAQ 化（検索意図に直結） |

## ヘッダの解釈（厳密固定しない）

ラッコの画面ごとに列は異なり、将来変わりうる。エージェントは以下の**意味**でヘッダを読む：

- 「キーワード」列 → 対象語
- 「月間検索数 / 検索ボリューム / Volume」列 → 需要の実数
- 「難易度 / SEO難易度 / KD」列 → 競合の強さ
- 「見出し / H2 / H3」列 → 競合構成
- 「共起語」列 → 網羅チェック用の語

**必要な列が見つからないときは、その旨を報告し、従来の推測（WebSearch/SERP精読）に切り替える。** 欠損を黙って埋めない。

## 使うときの原則

- **データは「加点」。** 無ければ従来フロー、あれば精度を上げる。データ待ちで停止しない
- **数値を鵜呑みにしない。** ボリューム・難易度はラッコの推定値。最終判断は SERP精読と併用する
- **共起語を詰め込まない。** 網羅チェックに使い、不自然な羅列にしない（詰め込みは editor が減点）
- **一次情報主義は不変。** 制度・数値の事実はラッコと無関係に公式で裏を取る（`blog.md`）
- **ラッコの AI 記事生成は使わない。** データのみ活用する（本文は writer が書く）
